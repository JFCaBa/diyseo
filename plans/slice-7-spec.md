# Slice 7 Architecture Plan: Manual Article Editor

## 1. Objective
Define the architecture for Slice 7, which introduces a manual article editor. This allows users to manually modify the fields generated by AI or create/edit articles entirely by hand.

## 2. Scope & Constraints
*   **Included**: UI for editing an article, Server Action for saving updates to existing fields (title, excerpt, contentHtml, seoTitle, seoDescription).
*   **Constraints**: No new database models or fields required (relies on Slice 4 fields). No changes to the public API or embeddable widget. No changes to AI logic. Keep the UI minimal and functional.

## 3. Database Usage
No schema changes are required. The editor will interact with the existing `Article` model, specifically targeting:
*   `title`
*   `excerpt`
*   `content` (mapped from `contentHtml` in UI)
*   `seoTitle`
*   `seoDescription`
*   `slug` (optional to allow manual override, but at minimum must not break uniqueness)
*   `updatedAt` (automatically handled by Prisma)

## 4. Internal API & Server Actions

We will use a Next.js Server Action to handle the update securely and trigger necessary revalidations.

### 4.1. `updateArticle(articleId: string, data: UpdateArticleInput)`
*   **Logic**:
    1.  Validate input data using Zod.
    2.  Verify the article exists and belongs to the current `siteId`.
    3.  Update the Prisma `Article` record with the provided fields.
    4.  Call `revalidatePath` for both the editor page and the main articles list to reflect changes immediately.

## 5. UI Behavior

### 5.1. Editor Access (`/[siteId]/articles/[articleId]`)
*   **Entry Point**: A simple "Edit" button or link on each article row in the `/[siteId]/articles` list.
*   **Layout**: A form containing inputs for:
    *   **Title**: Standard text input.
    *   **Excerpt**: Textarea.
    *   **SEO Title**: Text input with character count indicator (max 60).
    *   **SEO Description**: Textarea with character count indicator (max 160).
    *   **Content**: For V1, a simple `<textarea>` for raw HTML/Markdown is acceptable, keeping it minimal and stable without integrating complex WYSIWYG editors yet.
*   **Actions**: "Save Changes" button, which triggers the Server Action. A "Back to Articles" link.

## 6. Validation Rules (Zod)

```typescript
import { z } from 'zod';

export const UpdateArticleSchema = z.object({
  title: z.string().min(1, "Title is required").max(200),
  excerpt: z.string().optional().nullable(),
  content: z.string().min(1, "Content is required"), // Maps to contentHtml
  seoTitle: z.string().max(60, "SEO Title should be under 60 characters").optional().nullable(),
  seoDescription: z.string().max(160, "SEO Description should be under 160 characters").optional().nullable(),
});
```

## 7. Acceptance Criteria
*   [ ] An "Edit" button exists on the Articles list page that navigates to the editor screen for a specific article.
*   [ ] The editor screen correctly populates its fields with the article's existing data from the database.
*   [ ] User can modify the title, excerpt, content, seoTitle, and seoDescription.
*   [ ] Clicking "Save Changes" successfully updates the article in the database via a Server Action.
*   [ ] Validation rules are enforced (e.g., character limits on SEO fields).
*   [ ] After saving, the user sees a success indication and the updated data is reflected on subsequent reloads.
